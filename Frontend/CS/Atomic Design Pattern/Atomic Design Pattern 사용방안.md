# Atomic Design Pattern 사용방안

### 정의

- [Atomic Design Pattern](https://bradfrost.com/blog/post/atomic-web-design/)은 2013년 6월, Brad Frost에 의해 처음으로 제시된 디자인 패턴입니다. 
- 컴포넌트들을 논리 구조 (Building Blocks) 단위로 나누어 효율적으로 재사용하도록 만들어서, 불필요한 "코드 중복"을 제거하고 "클린"한 아키텍쳐를 유지할 수 있도록 합니다. 
- 이 패턴은 디자인 시스템(재사용을 위해 만든 컴포넌트의 집합)을 위해 만들어진 패턴입니다. 



### 설계 원리

![image-20221227235637063](C:\Users\sell\AppData\Roaming\Typora\typora-user-images\image-20221227235637063.png)

- HTML 태그에 가까운 ATOM을 만듭니다. 
- ATOM을 이용하여 조금 더 복잡한 MOLECULES를 만듭니다.
- ATOM과 MOLECULES를 이용하여서 ORGANISMS을 만듭니다.
- ORGANISMS를 이용하여 TEMPLATES을 구성합니다.
- TEMPLATES 을 제외한 공간에 organism 이하 컴포넌트를 배치함으로서 PAGE를 완성합니다.



### 문제점 : 잘못된 사용법

- 하지만, 아토믹 디자인 패턴을 제대로 이해하지 못하고, 컴포넌트를 쪼개는 것에 집착하여 구성하다보면 역으로 코드의 가독성과 유지보수가 나쁜 아키텍쳐를 만듭니다.

- 이는 컴포넌트를 디자인 시스템의 정의대로 기능으로 나눈 것이 아니라 뷰의 단위로 나누기때문에 발생합니다.

  ```
  논리적인 단위로서의 버튼(Atoms)은 "클릭하면 바인딩된 특정 액션을 실행한다"라는 하나의 기능만을 제공하므로 그것이 가지고 있는 style(hover, active....)과는 상관없이 하나의 기능을 수행하는 하나의 단위입니다
  
  논리적인 단위로서의 검색바(Molecules)는 "클릭하면 특정 액션을 실행하는" 검색 버튼(Atom)과 "타이핑하면 글자가 입력되는" Input(Atom)으로 이루어져있지만, 이 둘이 합쳐져서 "타이핑해서 클릭하면 해당 글자로 검색을 수행하는" 하나의 기능을 수행하는 하나의 단위입니다
  ```

- 위 예시와 달리, 기능이 아닌 뷰를 기준으로 컴포넌트를 분류할 경우, 페이지가 많아질수록 재사용과 관리가 어려운 컴포넌트가 많아지고, 관리가 힘들어집니다.



### 해결방안

- 뷰가 아닌 기능단위로 분류합니다. 이 때, props를 통해서 넘기는 style 요소(css) 등은 최대한 생략할 수 있게 설계하고 기능적으로 집중할 수 있게합니다. 
- 만약 특정 컴포넌트에 종속성이 있는 하위 컴포넌트나 레이아웃의 집중하여 컴포넌트를 만들어야한다면, 이는 공통컴포넌트(atomic)로 두지 않고 개별관리합니다.



### 고찰

- 이 패턴에 대해서 실제로 적용해본 분들에 이야기에 따르면, 멘탈적 모델로서 훌륭하지만 실무로서는 별로라는 분이 많았다.
- 주로 협업에서 굉장히 큰 트러블을 느끼지만, 개인의 실력 향상에는 큰 도움이 되었다고한다.
- 나도 이를 개인적으로 사용하는 중인데, 생각보다 다양한 생각을 하게된다. 
  - 아 이 부분은 공통으로 제작할 수 있겠다. 왜 괜히 컴포넌트를 여러개 만들고 있었지
  - 이 부분은 구조상 유기체인지 분자인지 좀 애매한대...
  - 기능단위로 컴포넌트를 만든다는 것이 생각보다 굉장히 어려운 일이구나 무의식적으로 뷰단위로 컴포넌트를 나누려한다.
  - props를 생각해서 설계한다는 것이 굉장히 많은 걸 생각하게 한다.
